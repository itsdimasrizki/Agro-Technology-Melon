Arsitektur Sistem

Master Node berfungsi sebagai pusat kontrol FSM yang membaca sensor kualitas air, mengelola relay pompa, dan menerima data tanah dari Sleeve Node via ESP-NOW. Tim Cloud akan menjembatani data ini ke cloud melalui protokol MQTT.

Struktur Data Sensor

Struct SensorData adalah data utama yang harus dipublikasikan ke MQTT setiap interval. Ini berisi seluruh pembacaan sensor dari Master Node ditambah data tanah dari Sleeve.

Field Tipe Unit Sumber Deskripsi
temperature float °C DS18B20 Suhu air di tandon nutrisi
ph float pH DFRobot pH Tingkat keasaman larutan (0–14)
ppm float ppm TDS Sensor Konsentrasi nutrisi (parts per million)
waterLevel float cm AJSR04M Ketinggian air dalam tandon
soilADC uint16_t ADC raw Sleeve (ESP-NOW) Kelembapan tanah mentah dari lahan
flowWater float Liter Flow Meter #1 Volume air baku yang terisi
flowA float Liter Flow Meter #2 Volume nutrisi A yang dipompa
flowB float Liter Flow Meter #3 Volume nutrisi B yang dipompa
currentVolume float Liter Kalkulasi Estimasi volume total larutan saat ini
📥 SoilData from Sleeve
Data yang diterima dari Sleeve Node via ESP-NOW. Saat ini hanya berisi satu field:
struct SoilData { uint16_t averageRawADC; // ADC kelembapan tanah (filtered) };
🧪 NutrientRecipe per Fase
Resep nutrisi yang berubah sesuai usia tanaman (HST). Digunakan FSM untuk menentukan target pencampuran.
struct NutrientRecipe { float targetPPM; // Target konsentrasi nutrisi float targetMinPH; // Batas bawah pH float targetMaxPH; // Batas atas pH };

Tabel State FSM

Detail setiap state dalam enum FertigationState. Kolom Aksi Relay menunjukkan relay mana yang aktif, penting untuk logging di cloud.

ID State Deskripsi Relay Aktif Timeout
0 IDLE Sistem nonaktif, semua relay OFF — —
1 WAIT_DAILY_MIX Menunggu jadwal harian (05:00) — —
2 PREPARE_DAILY_MIX Memuat resep nutrisi berdasarkan usia tanaman — —
3 FILL_WATER Mengisi air baku ke tandon (160L) RELAY_WATER 30 menit
4 ADD_NUTRIENT_A Memompa nutrisi A (0.8L) RELAY_NUTRIENT_A 5 menit
5 MIX_A Pengadukan setelah nutrisi A — 5 menit (fixed)
6 ADD_NUTRIENT_B Memompa nutrisi B (0.8L) RELAY_NUTRIENT_B 5 menit
7 MIX_B Pengadukan setelah nutrisi B — 5 menit (fixed)
8 VALIDATE Cek PPM apakah sudah sesuai target resep — —
9 CORRECT_PPM Koreksi PPM: tambah A+B masing-masing 50mL RELAY_A + RELAY_B —
10 CORRECTION_MIX Pengadukan koreksi sebelum re-validasi — 1 menit (fixed)
11 READY Larutan siap, menunggu tanah kering — —
12 IRRIGATION Irigasi aktif sampai tanah cukup basah RELAY_IRRIGATION —
13 ERROR Error: semua relay OFF, butuh reset manual — (all OFF) —

Panduan Integrasi MQTT

Berikut adalah panduan untuk tim Cloud & MQTT dalam mengintegrasikan firmware Master Node ke infrastruktur cloud. Sistem saat ini menggunakan ESP-NOW untuk komunikasi lokal, dan akan ditambahkan lapisan MQTT untuk mengirim data ke broker.

⚠️ Penting: Wi-Fi vs ESP-NOW
ESP-NOW dan Wi-Fi bisa berjalan bersamaan pada ESP32-S3, namun harus di channel yang sama. Pastikan Wi-Fi station mode terhubung terlebih dahulu, lalu ESP-NOW mengikuti channel Wi-Fi. Urutan inisialisasi sangat krusial!

📤 Titik Integrasi Publish
Tempat di kode firmware dimana MQTT publish harus disisipkan:
// 1. Di loop() — Main.ino (setiap 1 detik) if(millis() - lastPrint >= 1000) { SensorData data = sensorManager.getData(); mqttPublishSensors(data); // ← tambahkan } // 2. Di changeState() — FertigationFSM.cpp void FertigationFSM::changeState(...) { state = newState; mqttPublishState(newState); // ← tambahkan } // 3. Di handleError() — FertigationFSM.cpp void FertigationFSM::handleError() { mqttPublishAlert("FSM_ERROR"); // ← tambahkan }
📥 Titik Integrasi Subscribe
Command dari cloud yang perlu didengarkan oleh Master:
// Callback MQTT yang perlu dibuat: void onMqttMessage(topic, payload) { if(topic == "agro/master/cmd/reset") { fsm.begin(); // Reset FSM ke awal } if(topic == "agro/master/cmd/irrigate") { // Manual override irigasi } if(topic == "agro/master/cmd/stop") { relay.allOff(); // Force ke IDLE } }

Rekomendasi MQTT Topic

Struktur topic MQTT yang direkomendasikan mengikuti pola hierarki: agro/{node}/{category}/{subcategory}

agro/master/sensors
Data semua sensor dalam format JSON, dikirim setiap 1 detik
Publish
agro/master/fsm/state
State FSM saat ini (integer 0-13), dikirim saat terjadi transisi
Publish
agro/master/fsm/error
Alert error dengan detail kode dan timestamp, retain = true
Publish
agro/master/recipe/current
Resep nutrisi yang sedang aktif (PPM target, pH range), dikirim saat PREPARE_DAILY_MIX
Publish
agro/master/relay/status
Status relay (bitmask 8-bit), dikirim saat on/off relay
Publish
agro/master/cmd/reset
Perintah reset FSM dari dashboard cloud
Subscribe
agro/master/cmd/irrigate
Manual override: paksa mulai irigasi dari dashboard
Subscribe
agro/master/cmd/stop
Emergency stop: matikan semua relay dan pindah ke IDLE
Subscribe

Format JSON Payload

Contoh payload JSON yang direkomendasikan untuk setiap topic. Gunakan format ini sebagai kontrak antara firmware dan backend cloud.

📤 agro/master/sensors QoS 0
{ "timestamp" : 1719100800, "temperature" : 27.5, "ph" : 6.2, "ppm" : 850, "waterLevel" : 45.3, "soilADC" : 2847, "flow" : { "water" : 155.2, "nutrientA": 0.78, "nutrientB": 0.80 }, "volume" : 156.78 }
⚙️ agro/master/fsm/state QoS 1 Retain
{ "state" : 3, "name" : "FILL_WATER", "since" : 1719100800, "plantAge" : 22 }
🚨 agro/master/fsm/error QoS 2 Retain
{ "error" : "WATER_FILL_TIMEOUT", "fromState" : "FILL_WATER", "timestamp" : 1719100800, "sensors" : { /* snapshot */ } }
✅ Tips Payload
• Selalu sertakan timestamp (Unix epoch) agar time-series DB bekerja optimal.
• Gunakan QoS 1 + Retain untuk state FSM agar subscriber baru langsung tahu state terakhir.
• Untuk sensor data yang frekuensinya tinggi (1 detik), gunakan QoS 0 untuk efisiensi bandwidth.
